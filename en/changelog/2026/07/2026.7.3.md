## ✨ New Features

### Checkpoint Node

New **Checkpoint** node to measure flow performance in an **explicit, non-blocking** way. Instead of trying to infer follow-ups automatically from the session timeout (which produced incorrect records), you now **place the checkpoint exactly where you want to measure** — after sending a message, before a branch, or at any step in the flow.

#### What is it for?

- Count **how many conversations pass** through a specific point in the flow (volume)
- Measure whether the customer **replies** within a configurable window after passing that point (response rate)
- Capture a **snapshot of flow variables** at the moment of passage for later analysis
- Compare performance **across different points** within the same flow

#### How to use it?

1. In the **Flow Builder**, open the **📊 Statistics** category in the node panel
2. Drag the **Checkpoint** node onto the canvas and connect it in the flow (input and output — the node does not block execution)
3. Click the node body to configure:
   - **Track passage (volume)** — records each passage as a volume event
   - **Track customer response** — starts the response window countdown; any customer message within the window marks the event as responded
   - **Response window** — fixed value in minutes or a numeric flow variable (default when creating the node: **48h / 2880 min**)
   - **Save flow variables** — optional; stores variable state at the moment of passage
4. **Publish** the flow so events start being recorded in production

> 💡 **Non-blocking**: the flow continues immediately to the next node, like a variable node. Registration failures never block the conversation.

> Full guide: [Checkpoint Node](/en/guide/flows/follow-up-response-rate)

### Follow-up Analytics Report

Refreshed **Follow-up Analytics** panel to track Checkpoint events, with filters, manual refresh, and visuals aligned with the design system.

#### What is it for?

- View response rate, passage volume, average response time, and evolution over time
- Filter by **date range**, **service channel**, and **assigned agent**
- Compare metrics **by node** (name + ID) or focus on a **single checkpoint**
- Get a summarized indicator on the **Sales Funnel** (CRM)

#### How to use it?

1. **Full flow report**: in the flow editor, click **Follow-up Analytics** in the top bar
2. **Single checkpoint report**: on the Checkpoint node itself, click **View report for this point**
3. Adjust filters and use the **refresh** button to reload data
4. In **CRM → Sales Funnel**, see the organization-wide summary card

> Full guide: [Checkpoint Node](/en/guide/flows/follow-up-response-rate)

## 🔄 Improvements

### Automatic tracking removed

The automatic tracker coupled to **"Wait for Response" timeouts** and **AI Agent reschedules** was removed — it produced incorrect records by conflating response timeouts with other internal flow mechanisms. As of this version, **only explicitly placed Checkpoints** generate metrics.

### Flow test using unpublished draft

When testing a flow directly from the **Flow Builder**, the test now uses your **draft** changes (not yet published), letting you validate changes before publishing. When a test is started from an **AI Agent** action (in Prompts), it keeps using the **published** version, as expected.

### More thorough test reset

When clicking **"Restart"** in the flow test, besides clearing the conversation messages, the system now fully removes the previous flow session linked to the chat — preventing leftover data from an old session from interfering with the new test.

### Notifications disabled during testing

Messages sent during a **flow test** no longer trigger push notifications to the team (assigned agent/collaborators), since it's a simulation and not a real conversation.

### Visual progress indicator during testing

While testing a flow in the Builder, the **edge** leading to the active node is now highlighted with an animation, making it easier to visually follow the conversation's path.

## 🐛 Bug Fixes

### Timeout checker performance

The routine that processes "Wait for Response" (Input node) timeouts was optimized: sessions are now processed in batches with controlled concurrency, and the check keeps running immediately until the pending queue is drained, instead of waiting for the next scheduled cycle. This eliminates delays of up to 20-30 minutes that could occur with large queues.

### Node incorrectly highlighted in the Flow Builder

Fixed an issue where more than one node could appear highlighted (green border) at the same time while testing a flow — even after reloading the page. The cause was leftover test data that had been incorrectly saved into the flow; affected flows have already been fixed, and an extra safeguard prevents this from happening again.
